Dear all, I would like to start a discussion about LightSheet Deconvolution. Which workflow do you use? From my point of view there is no solution that covers all modalities. Especially the excitation PSF from the different systems is not calculated optimally. I use Hyugens and the Lighsheet Deconvolution in ZEN Blue. Both are not optimal. In my opinion, the left and right light sheet should be deconvolved separately (with the scatter paramaters ) and then blending is done based on the decon data. The decon in ZEN Blue can only handle already processed light sheets (Max or DFT)) which is not the best way and Huygens can handle left and right separately but then it lacks a reasonable solution for blending both data sets. The BigSticher has a very good solution to use the left and right light sheets (depending on the position in the dataset) but it lacks a reasonable deconvolution. You see a huge topic.....

Well, for me these "optimizations" in BigStitcher are not really optimized for lightsheet data. If we use data sets with left and right light sheet separated then we have the illumination selection in BigStitcher already in place. Here, depending on the global position in the dataset, it is decided whether the left or the right light sheet is used during the tile stitching. An adaptive deconvolution would be the best , which takes into account the scatter properties of the left and right light sheet per tile. If this is implemented in the BigStitcher deconvolution approach then it would be indeed a good solution. I guess here simply one excitation PSF is used per tile over the whole FOV (just the classic approach).

*Thread Reply:* Hi Ren, I've been wondering the same thing. My users often want to analyse their data, but they don't have the computing resource to load the fully stitched, full resolution dataset. I guess one question is whether you need the full bit depth and resolution of your images? A 2x resolution downsample in 3D is 8x less data, and 8-bit is half of 16-bit. I'm sure these are things you've already considered.

If you can output some kind of transform file that describes the stitch, maybe you could carry out processing on each of the tiles and then stitch afterwards using the transform file (assuming you've stitched in the first place).

You could of course design your own analysis code, depending on how in-depth you want to go. But it looks like cellpose can segment in chunks, found this from the image.sc forums here.

I'm watching this thread to see what advice people have 🙂 - might be worth posting on the image.sc forum instead for this particular question - as your question applies to most types of data, not just light-sheet.
